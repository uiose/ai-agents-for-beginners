## 2. 先给这个项目一个准确定位

先用一句话定义它：

> `course_qa_agent.py` 是一个基于课程资料的单 Agent 问答原型，它使用大语言模型作为推理核心，使用本地课程文档作为知识来源，使用检索函数作为工具，通过 Microsoft Agent Framework 组织成一个可以多轮交互的最小闭环。

这个定位很重要，因为它说明它不是下面几类东西：

- 它不是一个普通的“读取 Markdown 再搜索关键词”的脚本

这和 `lab/README.md` 里的定位是完全一致的：单 Agent、本地检索增强问答、带简单工具调用，重点是先验证最小闭环。

## 3. Lesson 1：从“什么是 AI Agent”来理解你的原型

### 3.1 课程里的核心定义

在课程第 1 课里，AI Agent 被定义为：

> AI 代理是一个系统，它通过赋予 LLM 访问工具和知识的能力，从而让模型能够执行操作。

这句话可以拆成四层：

1. 它是一个系统，不只是一个模型调用
2. 它有 LLM，负责理解用户问题和组织回答
3. 它有工具，负责扩展模型能力
4. 它有知识来源，负责让回答不只依赖模型自身参数

你的 `course_qa_agent.py` 恰好把这四层都具备了。

### 3.2 代码里分别对应什么

#### 1. 系统

你的代码不是只写了一句“把问题发给模型”，而是组织了一个完整运行流程：

- 加载知识库
- 清洗文本
- 分块
- 检索
- 创建 Agent
- 创建会话
- 处理单次问答或多轮问答

这说明它已经是一个小型系统，而不是单次 API 调用。

#### 2. LLM

`OpenAIChatClient` 是模型接入层。  
它负责把用户问题交给模型，让模型基于指令和工具来生成响应。

在这一层，模型的职责不是“硬编码检索”，而是：

- 理解用户意图
- 判断要不要调用工具
- 在拿到工具结果后整合成自然语言答案

#### 3. 工具

你的 Agent 注册了两个工具：

- `retrieve_course_context(query, top_k=3)`
- `list_available_sources()`

这两个函数不是普通辅助函数而已，一旦被注册到 Agent，它们就变成了模型可用的外部能力。

这和你在 `ch01-基本agent.md` 里总结的机制完全一致：

> 模型负责决定调用什么，框架负责真正执行它。

#### 4. 知识

你的知识来源不是模型参数本身，而是 `DEFAULT_DOC_PATHS` 指向的课程资料：

- `02-explore-agentic-frameworks/README.md`
- `03-agentic-design-patterns/README.md`
- `04-tool-use/README.md`
- `azure-ai-foundry-agent-creation.md`

这正对应课程里“Agent 不只依赖模型，还可以访问外部知识”的定义。

### 3.3 为什么它确实算 Agent，而不是普通问答脚本

如果它只是下面这种流程：

1. 读取文档
2. 做关键词搜索
3. 把搜索结果打印出来

那它更像“本地检索工具”。

如果它只是下面这种流程：

1. 用户提问
2. 直接把问题扔给模型
3. 模型自由回答

那它更像“普通聊天机器人”。

而你的实现是：

1. 模型接收用户问题
2. 模型被系统指令要求优先检索
3. 模型可以调用工具获取课程上下文
4. 框架执行工具
5. 模型根据工具返回内容来组织答案
6. 会话保存上下文，支持连续交互

这就是一个标准的最小 Agent 闭环。

### 3.4 它更接近课程里哪类 Agent

按照第 1 课的分类，你这个原型最接近：

- **目标驱动 Agent**：目标是“回答课程相关问题，并给出可追溯依据”
- 同时带有一点**工具型 Agent**的特征：遇到需要依据的问题时，不直接空想，而是借助检索工具

它暂时还不是：

- 学习型 Agent，因为它没有根据反馈更新策略
- 分层 Agent，因为没有上层/下层代理拆分
- 多 Agent 系统，因为当前只有一个 Agent

所以你可以很稳地说：

> 我现在做的是一个单任务、单 Agent 的课程资料问答 Agent，它已经符合课程中对 Agent 的基本定义，但还没有扩展到多 Agent 或长期学习能力。

## 4. Lesson 2：从“为什么要用 Agent Framework”来理解你的实现

### 4.1 课程第二课在讲什么

第 2 课强调，Agent Framework 的价值不只是“帮你调模型”，而是把构建 Agent 时经常重复出现的组件组织起来，例如：

- 模型连接
- 工具注册
- 会话管理
- 多轮交互
- 代理行为封装

这节课最核心的思想是：

> 框架不是替你思考业务，而是替你管理 Agent 的基础设施。

### 4.2 你的代码如果不用框架，会发生什么

如果不用 Microsoft Agent Framework，你得自己手动做很多事：

1. 自己定义工具 schema 给模型
2. 自己把工具说明和对话消息组装成请求
3. 自己解析模型返回的 tool call
4. 自己调用本地函数
5. 自己把工具结果重新塞回模型上下文
6. 自己管理多轮消息历史

也就是说，真正麻烦的不是“写一个检索函数”，而是把“模型、工具、上下文”串成一个稳定流程。

### 4.3 你这里用了框架之后，得到了什么

在你的实现里，`create_agent()` 做了三件非常关键的事：

1. 创建模型客户端
2. 定义 Agent 的系统指令
3. 注册 Agent 的工具

然后在 `run_single_query()` 和 `run_interactive()` 里，框架继续帮你处理：

- 会话创建
- 消息流转
- 工具调用
- 返回响应结构

换句话说，你写的业务逻辑集中在：

- 课程资料如何组织
- 本地检索怎么做
- Agent 应该遵守什么回答规则

而不必自己实现一整套 tool-calling runtime。

### 4.4 这和课程中的框架能力是怎么对应的

第 2 课讲到，Agent Framework 提供的核心价值包括：

- 模块化组件
- 工具定义和代理管理
- 会话状态维护
- 更快的原型验证

你的代码几乎逐条体现了这些点：

#### 模块化组件

你把逻辑拆成了清晰的函数：

- 文本清洗：`clean_text`
- 关键词提取：`extract_terms`
- Markdown 分块：`iter_markdown_chunks`
- 语料加载：`load_corpus`
- 检索评分：`score_chunk`
- 检索入口：`retrieve_course_context`
- Agent 创建：`create_agent`

这让整个原型非常适合继续迭代。

#### 工具定义和代理管理

你不是把检索逻辑写死在主流程里，而是把它包装成 Agent 工具。  
这是从“程序思维”走向“Agent 思维”的一个关键变化。

#### 会话状态维护

在交互模式下，你通过 `agent.create_session()` 保留了连续对话上下文。  
这正对应课程里提到的“多轮交互中的状态管理”。

#### 更快的原型验证

你的 `lab/README.md` 已经明确表达了这个目标：先做一个能跑通、能展示、能汇报的最小闭环。  
这一点其实非常符合第 2 课的精神：框架让团队更快从概念走到原型。

### 4.5 为什么这里选单 Agent 是合理的

第 2 课也提到多 Agent 协作，但你当前没有直接做成多 Agent，这反而是合理的。

原因是：

- 你的当前任务边界非常明确：课程资料问答
- 工具类型也比较单一：检索和来源查询
- 还处在验证闭环阶段，不适合过早引入多 Agent 编排复杂度

所以你现在的选择可以解释为：

> 先用框架验证“一个 Agent + 一组工具 + 一套知识范围”是否能跑通，再决定是否有必要拆成检索代理、总结代理、评估代理等多个角色。

这是一种很稳健的工程路径。

## 5. Lesson 3：从 Agent 设计原则看这个原型为什么“收得住”

### 5.1 第三课不是教你“多做功能”，而是教你“怎么做得可信”

第 3 课最重要的地方，不是引入更多技术名词，而是强调：

- 透明性
- 控制权
- 一致性

这些原则对于你的原型非常有帮助，因为你的目标不是“看起来很聪明”，而是“回答尽量可靠，适合学习和汇报”。

### 5.2 透明性：你的原型已经做了哪些

透明性的核心是让用户知道：

- AI 正在做什么
- 它是基于什么回答的
- 它的边界在哪里

你的实现里已经有几处很好的透明性设计：

#### 1. 要求回答结尾给出来源

在 `AGENT_INSTRUCTIONS` 中，你明确要求：

- 优先调用检索工具
- 只基于检索上下文回答
- 结尾给出“来源”

这不是风格问题，而是可信性的核心设计。

#### 2. 知识范围是显式限定的

你没有让 Agent 假装“什么都知道”，而是通过 `DEFAULT_DOC_PATHS` 把知识范围收缩到几份课程材料。  
这使它具备清晰的能力边界。

#### 3. `--show-context` / `--preview` 提供了可观察性

这两个能力非常像课程里说的“让用户知道系统是如何工作的”。

- `--preview` 可以只看本地检索结果
- `--show-context` 可以先看检索上下文，再看模型回答

这意味着你不是把系统做成一个完全黑盒。

### 5.3 控制权：这个原型目前有哪些体现

课程里说，用户应当对系统有适当控制。  
在你的原型里，这种控制还比较基础，但已经开始出现：

#### 1. 用户可以决定使用哪种模式

- 单次问答
- 交互模式
- 只看检索不调用模型
- 先看上下文再调用模型

这让用户不是被动接受一个固定流程。

#### 2. 用户可以通过问题形式影响检索和回答

你的 Agent 不是完全固定输出，而是根据用户提问方向来选择相关资料片段。  
这是一种最基本的交互控制。

### 5.4 一致性：为什么你的 CLI 形式反而是优点

课程里讲一致性时，更多举的是 UI/UX 例子，但放到你这个原型里，可以理解为：

- 输入方式稳定
- 输出结构稳定
- 行为规则稳定

你的命令行交互其实很统一：

- 入口清晰
- 模式清晰
- 响应结构清晰

尤其是“先检索、再回答、附来源”的输出规范，非常适合学习场景。

### 5.5 这个原型在设计上为什么是“收得住”的

很多初学 Agent 的项目容易一开始就做得过大，比如：

- 想做通用智能助手
- 想接十几个工具
- 想做多 Agent 自动协作
- 想连外部数据库、联网检索、语音输入一起上

你的原型反而做对了一件很重要的事：

> 主动收缩任务边界。

从第 3 课的设计原则来看，这种收缩是有价值的，因为它：

- 降低了不确定性
- 增强了可解释性
- 让能力边界更清晰
- 更利于做可信输出

所以你可以说：

> 我不是暂时没做大，而是先按照可信 Agent 的设计思路，把任务压缩到一个可以解释、可以验证、可以展示的范围里。

### 5.6 目前还可以继续补强的地方

从第 3 课角度看，这个原型还可以继续增强：

- 更明确地提示当前知识范围
- 对“资料不足”场景给出更显式反馈
- 为回答模板增加统一结构
- 增加失败样例记录
- 增加用户可控的 `top_k`、知识源开关等参数

这些改进都不需要推翻现有架构，只是在现有基础上把可信性做得更完整。

## 6. Lesson 4：从工具使用设计模式拆解这个 Agent 的真正闭环

### 6.1 第四课的核心问题

第 4 课讲的是：

> 工具使用设计模式，就是让 LLM 能借助外部函数或服务去完成它本来做不到、或不适合仅靠模型完成的事情。

你这个项目正是一个很典型的工具使用案例，因为“根据课程资料回答并附依据”这件事，如果只靠模型记忆，很容易：

- 回答泛化
- 缺少出处
- 混入课程外信息

所以你才需要工具。

### 6.2 你的工具到底是什么

你当前的 Agent 注册了两个工具：

#### 工具 1：`retrieve_course_context`

这是核心工具。  
它做的事不是直接回答问题，而是：

1. 接收用户问题
2. 从本地语料里找最相关的片段
3. 把片段连同来源、章节、内容一起返回给模型

这个函数相当于告诉模型：

> 如果你想回答课程问题，先别凭空答，先拿资料。

#### 工具 2：`list_available_sources`

这个工具更像一个辅助元工具。  
它帮助 Agent 了解“自己有哪些资料可查”，也能在用户问“你查了哪些文件”时提供边界说明。

这虽然简单，但其实很符合课程里“工具不仅是执行动作，也可以帮助代理理解环境”的思想。

### 6.3 工具使用设计模式中的关键部件，在你的代码里分别是什么

第 4 课里提到几个关键构建模块，你的代码都有对应：

#### 1. 函数 / 工具 schema

你没有手写 JSON schema，但通过把 Python 函数注册到 Agent，框架会把这些函数转成模型可理解的工具描述。  
也就是说，schema 这层工作由框架承担了。

这正是框架价值的一部分。

#### 2. 函数执行逻辑

这部分是你自己写的，也是你的核心业务逻辑：

- `clean_text()`：清理原始文本
- `extract_terms()`：提取查询关键词
- `iter_markdown_chunks()`：把 Markdown 切成更适合检索的段落
- `score_chunk()`：按关键词命中程度打分
- `retrieve_top_chunks()`：返回最相关片段

这一层决定了你的工具“拿什么内容给模型”。

#### 3. 消息处理系统

在你的原型中，这部分主要体现在：

- `run_single_query()`
- `run_interactive()`
- `extract_text_from_response()`

它们负责组织用户输入、Agent 响应和显示格式。

#### 4. 状态管理

状态管理主要由 `session` 实现。  
这意味着在交互模式里，Agent 不会把每个问题都当成完全独立的问题，而是保留短期上下文。

### 6.4 把一次完整问答流程按工具调用链展开

如果按课程第 4 课来讲，你这个 Agent 的一次问答可以拆成下面这个闭环：

1. 用户输入一个问题
2. Agent 接收到问题和系统指令
3. 模型看到自己可用的工具
4. 模型根据问题判断是否调用 `retrieve_course_context`
5. 框架执行这个函数
6. 检索函数从课程资料中找出最相关片段
7. 工具把“来源 + 章节 + 内容”返回给模型
8. 模型基于这些内容组织最终回答
9. 回答按照指令附上来源

这就是你在 `ch01-基本agent.md` 里总结的那条经典链路：

> 用户提需求 -> 模型选工具 -> 框架执行 -> 模型整理结果

### 6.5 为什么你的检索工具不是“外挂”，而是 Agent 的一部分

一个常见误解是：  
“我不就是写了个搜索函数，再让模型用一下吗？”

其实不是。

当检索函数被注册为 Agent 工具之后，它就不再只是一个独立脚本，而变成了 Agent 决策过程的一部分：

- 它影响模型能看到什么信息
- 它影响最终回答的依据质量
- 它限制回答边界
- 它改变了模型的行为方式

也就是说，这不是“模型 + 一个外部小函数”的松散拼接，而是“一个通过工具扩展了知识访问能力的 Agent 系统”。

### 6.6 你的实现为什么是“简化版 RAG”，但还不是标准生产 RAG

你的原型已经具备了 RAG 的核心思想：

- 先检索
- 再基于检索结果生成

但它目前仍然是简化版，而不是生产级标准 RAG，因为它：

- 用的是本地文件而不是专门的检索服务
- 用的是规则型打分而不是向量检索
- 没有单独的索引构建流程
- 没有召回评估、重排和效果评测

这并不是缺点，而是阶段选择。

更准确的表述是：

> 它已经验证了“课程问答 + 检索增强 + 工具调用”的最小可行路径，后续可以自然升级为 Azure AI Search 或更标准的 Agentic RAG。

## 8. 这个原型已经验证了什么

## 9. 当前原型的边界和不足

为了讲得更真实，也需要明确它目前还没有解决什么。

### 9.1 检索能力仍然偏简单

当前主要依赖：

- 关键词提取
- 文本包含匹配
- 规则打分

这适合课程原型，但在更复杂场景下可能出现：

- 召回不稳定
- 语义匹配不够强
- 问法变化后效果波动

### 9.2 知识库范围是刻意受限的

这有助于保证可控性，但也意味着：

- 它无法回答课程范围外问题
- 也无法覆盖仓库中的所有内容

### 9.3 还缺少显式评测

目前更多是“能跑、能答、能展示”，还不是“经过系统评估证明准确率稳定”的状态。

### 9.4 还不是部署型 Agent

它目前更像本地实验原型，还没有：

- Azure AI Search 集成
- Azure AI Agent Service 部署
- 可视化前端
- 生产级日志与监控

这些不足不是问题，而是你下一阶段最自然的扩展空间。

## 11. 如果你明天要汇报，可以直接这样讲

下面这段表述可以直接作为口头汇报提纲：

### 11.1 项目定义

我目前做的是一个课程资料问答 Agent，不是通用助手。  
它的目标很明确，就是围绕课程内容回答问题，并尽量给出依据。

### 11.2 为什么它算 Agent

因为它不是单次调用模型，而是把模型、工具、知识和会话组织成了一个系统。  
模型负责理解问题，检索工具负责从课程资料里找上下文，最后再由模型基于这些上下文生成回答。

### 11.3 为什么要这么做

因为如果只靠模型直接回答，容易泛化、缺乏依据。  
所以我先把范围收缩到几份课程材料，用本地检索增强来提高可解释性和可靠性。

### 11.4 现在已经做到什么

当前已经完成了一个最小闭环：

- 单 Agent
- 本地课程知识库
- 简单工具调用
- 支持多轮会话
- 回答附来源

### 11.5 下一步做什么

下一步会考虑把本地检索升级成更标准的 RAG，比如 Azure AI Search，并补充评测样例，让这个原型从“能跑”逐步走向“更稳”。
