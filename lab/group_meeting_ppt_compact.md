# 组会汇报精简版

这份文档是给明天组会直接用的压缩版，不改动原来的 `group_meeting_ppt.md`。

## 第 1 页：研究目标

页面要点：

- 任务场景：课程资料问答辅助 Agent
- 输入输出：用户问题到带来源回答
- 当前目标：先验证最小可运行闭环

口头稿：

我目前先把 Agent 的问题收缩成一个比较具体的场景，就是课程资料问答。  
输入是用户提出的问题，输出是结合课程资料生成的回答，并尽量给出来源。  
现阶段我的重点不是直接做复杂系统，而是先把最小闭环跑通，确认这个方向是可行的。

## 第 2 页：方法设计

页面要点：

- 技术路线：大模型 + 本地检索增强 + 简单工具调用
- 框架实现：Microsoft Agent Framework
- 约束策略：先检索，再回答，并输出来源

口头稿：

当前原型采用的是单 Agent 架构。  
整体技术路线是大模型加上本地检索增强，再配合简单工具调用。  
我用 Microsoft Agent Framework 搭建了主流程，并要求 Agent 在回答前优先检索相关资料，尽量基于检索结果组织答案，同时在结尾给出来源。

## 第 3 页：当前实现

页面要点：

- 已接入 4 份中文课程资料
- 已实现检索预览、单次问答、交互式问答
- 已支持来源输出

口头稿：

目前我已经完成了一个最小原型，代码放在 `lab/course_qa_agent.py`。  
它默认接入了 4 份课程资料，包括 Agentic Framework、Design Patterns、Tool Use 和 Azure AI Foundry agent creation 相关内容。  
现在已经支持检索预览、单次问答和交互式运行，也就是说从用户提问到检索再到回答这条链路已经跑通了。

## 第 4 页：初步结果

页面要点：

- 成功案例：工具使用设计模式，回答稳定，来源清晰
- 一般案例：Framework 和 Service 的区别，能回答但不够细
- 边界案例：Foundry 创建 agent 关键元素，结果还较粗糙

口头稿：

我做了几条代表性问题的初步测试。  
其中“什么是工具使用设计模式”这个问题效果最好，系统能够比较稳定地命中文档并给出来源。  
但对于“Framework 和 Service 的区别”这类对比问题，以及“Foundry 中创建 agent 的关键元素”这类流程性问题，当前回答还偏概括或者不够完整，这也说明系统还有比较明显的改进空间。

## 第 5 页：问题与下一步

页面要点：

- 当前问题：检索精度还不稳定
- 当前问题：回答粒度和可解释性还不够
- 下一步：升级为更标准的 RAG 并补评测样例

口头稿：

当前最主要的问题有两个。  
第一是检索层还比较轻量，面对抽象问题时相关段落排序不够稳定；第二是虽然已经能输出来源，但回答粒度和证据组织还不够好。  
下一步我准备把现在的本地检索升级成更标准的 RAG 流程，比如接入 Azure AI Search 或向量检索，同时补一个小规模测试集，系统评估回答质量。

## 30 秒结束语

我现在的工作重点是先把任务收缩清楚，并验证最小原型是否成立。  
目前已经完成了课程资料问答 Agent 的基本闭环，也初步看到了它在哪些问题上有效、又会在哪些问题上出错。  
接下来我会继续优化检索和评测，把它逐步往更标准的 RAG Agent 方向推进。
