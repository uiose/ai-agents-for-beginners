## 一、对原问题的专家回答

### 1. 工具是代理在执行，而不是 LLM 在执行吗？

是。

这个问题的标准答案必须说得更精确一些：

- `LLM` 负责生成“是否调用工具、调用哪个工具、参数为何”的结构化意图
- `Agent Runtime` 负责解释这个意图，并决定是否真的执行
- `Tool Executor` 负责与数据库、搜索、文件系统、HTTP API 等外部能力交互
- 执行结果回流到 `LLM`
- `LLM` 再基于结果生成最终自然语言答复

因此，`LLM` 并不直接“执行工具”。  
它本质上只是在生成一份“动作建议”或“函数调用草案”。

这是理解 Agent 的第一道分水岭：

- 只把模型当作“会说话的程序”，会误解工具调用
- 把模型看成“决策组件”，才会真正理解 Agent Runtime 的必要性

### 2. 代理是不是必须跑在 CLI 环境里？

不是。

这是一个典型的“把演示壳层误认为系统本体”的认知偏差。

`CLI` 之所以常见，只是因为它适合教学和调试，便于观察：

- 用户输入
- 模型回复
- 工具调用
- 工具结果回填

但 `CLI` 从来不是代理能力成立的必要条件。  
代理可以运行在任何能承载请求处理和工具执行闭环的宿主环境中，例如：

- Web 后端
- 桌面应用本地进程
- 移动应用的云端服务
- Serverless
- 容器化微服务

真正的必要条件不是 `CLI`，而是一个可靠的运行闭环：

1. 接收请求
2. 组织上下文与工具 schema
3. 调用模型
4. 接收模型产出的工具调用
5. 执行工具
6. 回传结果
7. 生成最终回复

只要这条链路存在，代理就成立；这条链路不存在，换成任何外壳都没用。
> 我的疑惑是工具需要运行环境，那么这个运行环境只是软件实现上的问题吗？与智能体设计层面不用考虑吗？
### 3. 如果集成到软件里，应该怎么做？

如果从专业系统设计角度回答，这个问题不能停留在“把工具接上去”。

正确答案是：必须分层。

一个可上线的工具使用系统，至少分为五层：

### 3.1 交互层

负责接收用户输入和呈现系统输出。

典型形态：

- Web UI
- 桌面应用
- IM / Chat 入口
- 内部业务门户

### 3.2 业务控制层

负责非 Agent 专属但生产必须具备的治理能力：

- 鉴权
- 租户隔离
- 配额与限流
- 审计
- 合规控制

这一层如果缺失，Agent 往往只是“能跑”，而不是“能用”。

### 3.3 Agent 编排层

这是 Agent 系统的核心，不是模型本身。

它负责：

- 维护会话上下文
- 向模型暴露工具 schema
- 接收模型的工具调用意图
- 执行调用前校验
- 组织多轮调用循环
- 处理工具结果注入

### 3.4 工具执行层

这层才是真正触达外部世界的层。

它负责：

- 查询数据库
- 请求搜索服务
- 调用第三方 API
- 访问文件系统
- 触发企业工作流

这一层必须有明确权限边界，不能把“模型想调什么”直接等同于“系统就执行什么”。

### 3.5 状态与观测层

这层经常被初学者忽略，但它决定系统是否可维护。

它至少应包含：

- 会话状态
- 工具调用日志
- 错误与重试记录
- 性能指标
- 成本指标
- 审计轨迹

没有这层，系统只能演示，无法稳定运营。

### 4. 课程里提到的模块，是怎么完成的？

课程列出的模块并不是零散概念，而是工具调用闭环中的职责切分。

#### 4.1 函数 / 工具 schema

这是模型侧的契约层。

它的职责是告诉模型：

- 系统有哪些工具
- 每个工具做什么
- 参数如何组织
- 返回值大致是什么结构

常见实现形式：

- JSON Schema
- SDK / 框架自动生成的元数据
- 装饰器或注解转换后的工具描述

这层写得差，模型就会更容易误调用、乱传参数、选错工具。

#### 4.2 函数执行逻辑

这是控制层，不是描述层。

它决定：

- 什么时候允许调用工具
- 是否需要先澄清用户意图
- 多个工具如何选择
- 是否允许连续调用
- 写操作是否需要审批

简单系统把这部分交给模型。  
成熟系统会额外引入：

- 路由器
- 规则引擎
- 审批中间件
- 风险控制器

#### 4.3 消息处理系统

这是流程层。

它串起：

- 用户消息
- 系统提示
- 模型输出
- 工具请求
- 工具结果
- 最终答复

本质上，它是一个消息状态机或事件流编排器。

#### 4.4 工具集成框架

这是基础设施层。

它的价值不是“让你少写几行代码”，而是把重复且容易出错的工作标准化，例如：

- 工具注册
- schema 转换
- 参数序列化
- 调用循环
- 结果回填
- 中间件挂载

框架的作用是提高一致性和工程速度，不是替代架构判断。

#### 4.5 错误处理与验证

这是可靠性层。

至少要覆盖：

- 参数缺失
- 类型错误
- 超时
- 工具不可用
- 上游服务异常
- 模型误调用
- 非幂等操作重复执行

如果这一层缺失，你得到的不是智能系统，而是一个脆弱的自动化脚本。

#### 4.6 状态管理

这是长期一致性层。

它解决的不是“模型能不能回答”，而是：

- 多轮上下文是否一致
- 历史工具调用是否可追溯
- 中间结果是否可复用
- 用户状态是否可持久化

没有状态管理，Agent 只能表现出短时聪明，无法表现出稳定性。

### 5. 这些模块是谁提出来的？

严格地说，这个问题不能期待一个单一作者答案。

工具使用设计模式的形成，至少来自三类来源：

#### 5.1 研究范式

这一层回答“为什么模型需要借助工具”。

代表性思路包括：

- ReAct
- Toolformer
- MRKL

它们共同推动了“推理不应等同于行动，模型需要借助外部能力”的认识。
>这些思路的代表材料是？
#### 5.2 平台接口能力

这一层回答“模型如何表达调用工具的意图”。

也就是今天常说的：

- function calling
- tool calling
- structured tool outputs

这是模型平台将研究范式产品化的结果。

#### 5.3 软件工程结构

这一层回答“系统怎样可靠地把工具调用跑起来”。

例如：

- schema 契约
- 消息编排
- 插件机制
- 错误处理
- 状态管理
- 审计与追踪

所以，今天看到的“工具使用设计模式”不是单点创意，而是：

`研究范式 + 模型接口 + 软件工程`

三者叠加后的产物。

## 四、继续深挖时，真正值得问的问题

如果你想把这一章学到工程级，而不是课程级，下面这些问题更有价值：

这些问题比“怎么调用一个工具”更难，但它们才真正区分：

- 教学 Demo

### 第三优先级：治理与风险

- 权限模型
- 审批机制
- 幂等性
- 回滚与补偿

### 第四优先级：评估体系

没有评估体系，Agent 优化会长期停留在拍脑袋阶段。
