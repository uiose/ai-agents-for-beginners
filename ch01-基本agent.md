这份笔记整理自围绕**course01** `Semantic Kernel` 示例 notebook 的一组讨论，目标是用尽量清晰的方式回答几个关键问题：

- 什么叫“定义 tool”
- agent、LLM、tool 分别负责什么
- 模型如何知道有哪些 tool 可用
- 模型如何决定是否调用 tool
- tool 多了以后为什么会带来上下文和成本问题

## 1. 从示例代码看：这是不是在构建 tool

示例代码核心结构如下：

```python
class DestinationsPlugin:
    @kernel_function(description="Provides a random vacation destination.")
    def get_random_destination(self) -> Annotated[str, "Returns a random vacation destination."]:
        ...
```

这个结构里有两个层次：

- `DestinationsPlugin` 是一个 `plugin`
- `get_random_destination()` 是一个被 `@kernel_function` 暴露出来的 `tool` 或 `function`

所以，更准确地说：

- 这段代码是在定义一个 `plugin`
- 同时把其中的方法注册成 agent 可调用的 `tool`

## 2. 整个 notebook 在做什么

这个 notebook 本质上是在演示一个最小 agent 流程：

1. 接入 LLM
2. 定义工具
3. 把工具注册到 agent
4. 用自然语言触发 agent
5. agent 视情况调用工具
6. 再把结果组织成自然语言返回给用户

可以压缩成一句话：

> 定义一个 agent，接入模型，定义工具，注册工具，然后让用户通过自然语言使用它。

## 3. 谁负责什么

这套机制里，职责是分开的：

- 模型负责“理解用户意图，并判断要不要调用工具”
- 框架负责“把工具信息提供给模型，并在需要时真正执行工具”
- 工具负责“完成某个具体能力”
- agent 是把这些部件组织起来的运行单元

一句最重要的话：

> LLM 做的是选工具，程序做的是执行工具。

这句话一旦理解清楚，很多 agent 相关困惑都会减少。

## 4. 模型怎么知道有 tool 可用

模型不是直接读取你的 Python 类，也不是自己“发现”代码里有工具。

真正发生的是：

1. 你在代码里注册了 tool
2. 框架把这些 tool 转成结构化描述
3. 在请求模型时，把这些描述和对话上下文一起发给模型

这些描述通常包括：

- tool 名称
- tool 的用途说明
- 参数 schema
- 返回值的语义

所以模型“知道有 tool”，不是因为它理解了你的本地代码，而是因为框架把工具说明书发给了它。

## 5. 模型如何决定是否调用 tool

模型是否调用 tool，通常依赖以下几类信息：

- 用户当前请求
- system / instructions 提示词
- tool 的名称和描述
- 参数是否匹配当前任务

例如：

用户说：

```text
帮我推荐一个旅行地点
```

模型会看到当前工具里有一个描述大致是“提供随机旅行目的地”的函数，于是判断这个工具和用户意图匹配，倾向于发起调用。

如果用户说：

```text
什么是旅行规划
```

这更像概念解释，模型可能直接回答，而不是调用工具。

所以“是否调用”不是硬编码规则，而是模型根据上下文做出的推断。

## 6. 一次 tool 调用的完整链路

一次典型调用链可以概括为：

1. 代码中注册 tool
2. 框架把 tool 的元数据传给模型
3. 模型返回“我要调用某个 tool”
4. 框架在本地执行对应函数
5. 执行结果回传给模型
6. 模型把结果整理成自然语言输出

关键点：

- 模型不会直接执行 Python
- 真正执行代码的是你的应用程序或 agent runtime

因此可以记成一句口令：

> 模型调用，框架执行，返回结果。

## 7. 为什么 tool 描述很重要

模型选择工具时，高度依赖工具的描述信息。

例如下面这段：

```python
@kernel_function(description="Provides a random vacation destination.")
```

这段描述相当于告诉模型：

- 这个函数是干什么的
- 在什么场景下该选它

如果描述写得模糊、不准确，或者与实际功能不一致，模型选择它的概率就会下降，甚至会误选别的工具。

所以 tool 设计不是只有“函数写出来”就够了，还包括：

- 名称是否清晰
- 描述是否准确
- 参数是否合理
- 粒度是否合适

## 8. tool 多了以后，为什么上下文会变大

这是一个非常实际的问题。

因为框架通常会把“当前可用工具清单”一并发给模型，所以工具越多：

- token 消耗越大
- 请求成本越高
- 延迟可能变高
- 模型选错工具的概率也会增加

原因很简单：

- 模型要读更多“能力说明书”
- 工具之间如果描述相近，会产生歧义
- 可选项太多，决策难度会上升

所以 tool 本身不仅是能力，也是一种上下文负担。

## 9. 成熟系统如何处理 tool 过多的问题

实际系统一般不会把所有 tool 无差别地一次性暴露给模型，而是做控制。

常见方法有：

### 9.1 只暴露当前场景需要的工具

例如：

- 旅行 agent 只给旅行相关工具
- 报销 agent 只给报销相关工具

而不是让一个 agent 同时拥有几十种互不相关的能力。

### 9.2 先做路由，再选工具组

先判断用户意图，再决定暴露哪一组工具。

例如：

1. 用户询问天气
2. router 判定是“出行信息”
3. 只加载天气、地图、航班相关工具

### 9.3 缩短工具描述

描述必须清楚，但不应该啰嗦。  
描述越长，token 越高，噪声越多。

### 9.4 用高层能力封装低层工具

不要把太多零碎函数直接暴露给模型。

例如，不一定要单独暴露：

- `search_flights`
- `search_hotels`
- `search_cars`

可以把它们组织成更高层的旅行能力，由程序内部继续拆分。

### 9.5 动态注入工具

不是在 agent 初始化时永久绑定所有工具，而是根据当前会话阶段按需加载。

### 9.6 先做工具检索

当工具数量很大时，可以先根据用户问题筛选最相关的几个工具，再把这小部分发给模型。

## 10. 学习这块内容时，应该重点学什么

如果要系统解决“模型为什么会调用 tool、它怎么知道 tool 存在”的疑惑，建议按下面顺序学习。

### 10.1 Tool Calling / Function Calling

这是最核心的基础。

需要先理解：

- 模型不会直接运行代码
- 模型只会提出“调用某工具”的请求
- 宿主程序才是真正执行工具的一方

### 10.2 Tool Schema

理解工具对模型来说长什么样：

- 名称
- 描述
- 参数
- 返回值语义

只要看懂 schema，就能理解模型为什么会选某个工具。

### 10.3 Agent Runtime / Orchestration

学习框架如何完成这些步骤：

- 注册工具
- 把工具信息传给模型
- 解析模型返回的 tool call
- 执行本地函数
- 把结果再交给模型

在当前上下文里，对应的框架就是 `Semantic Kernel`。

### 10.4 ReAct / Agent Loop

学习 agent 的基本闭环：

1. 理解问题
2. 决定动作
3. 调用工具
4. 观察结果
5. 继续回答

这能帮助你从“单次 function calling”过渡到“完整 agent 思维”。

## 11. 一个最值得记住的抽象

可以把 agent 系统抽象成下面这张文字图：

```text
用户自然语言请求
        ↓
agent 收到请求
        ↓
框架把可用 tools 的说明发给 LLM
        ↓
LLM 判断：直接回答，还是发起 tool call
        ↓
如果发起 tool call：
    框架执行本地函数
    把结果返回给 LLM
        ↓
LLM 组织最终自然语言答案
        ↓
返回给用户
```

再压缩一次：

```text
用户提需求 -> 模型选工具 -> 框架执行 -> 模型整理结果
```

## 12. 最后的结论

围绕当前 notebook，可以得到以下结论：

- 代码中的 `plugin` 是工具集合
- `@kernel_function` 标记的方法才是模型可调用的 `tool`
- notebook 的整体目标是构建一个最小可工作的 agent 流程
- 模型知道 tool 的原因，是框架把 tool schema 发给了模型
- 模型是否调用 tool，取决于用户请求、提示词和 tool 描述之间的匹配程度
- tool 太多会导致上下文膨胀、成本上升和选错概率增加
- 成熟 agent 系统通常会做工具筛选、路由和动态加载，而不是一次性暴露全部能力

如果只保留一句最核心的话，那就是：

> 模型负责决定调用什么，框架负责真正执行它。
