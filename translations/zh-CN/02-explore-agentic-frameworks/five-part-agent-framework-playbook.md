# AI 代理框架的五部分主线：从概念到落地

本笔记专门回答一个常见问题：

- 我知道框架能“简化开发”，但它到底是怎么发挥作用的？
- 在“上下文、工具、记忆、规划、反馈闭环”这五部分里，AutoGen、Semantic Kernel、Azure AI Agent Service 分别做了什么？

你可以把 AI 代理系统看成一条持续运行的任务链：

1. 读取并组织上下文
2. 判断是否调用工具
3. 写入/检索记忆
4. 生成并执行规划
5. 用反馈闭环修正下一轮行为

---

## 1. 先建立统一心智模型：五部分各自解决什么问题

### 1.1 上下文（Context）

定义：让模型在“当前回合”知道哪些信息最关键。

关注点：

- 当前用户目标
- 当前任务状态
- 约束条件（权限、预算、时间、规则）
- 最近几步执行结果

一句话：上下文决定“这次该想什么”。

### 1.2 工具（Tools）

定义：让代理具备“执行动作”的能力，而不是只生成文本。

常见工具类型：

- 检索类：搜索、RAG、数据库查询
- 执行类：代码执行、工作流触发、函数调用
- 业务类：CRM、工单、日历、支付、库存系统

一句话：工具决定“这次能做什么”。

### 1.3 记忆（Memory）

定义：让代理跨回合保留有价值信息，减少重复提问并提高一致性。

常见分层：

- 短期记忆：会话内状态
- 长期记忆：用户偏好、历史事实、可复用经验

一句话：记忆决定“下次还记得什么”。

### 1.4 规划（Planning）

定义：把目标拆成步骤，并安排执行顺序与条件。

常见能力：

- 任务分解
- 步骤编排
- 条件分支
- 失败重试与回滚

一句话：规划决定“先做什么后做什么”。

### 1.5 反馈闭环（Feedback Loop）

定义：根据结果评估来修正下一轮策略，让系统持续改进。

典型信号：

- 用户反馈（满意/不满意）
- 规则评估（是否越权、是否合规）
- 业务指标（成功率、时延、成本）

一句话：反馈闭环决定“以后会不会变得更好”。

## 2. 三个框架在五部分中的角色分工

> 结论先行：
>
> - AutoGen：更擅长多代理协作与规划实验
> - Semantic Kernel：更擅长工具和记忆的工程化编排
> - Azure AI Agent Service：更擅长托管运行、线程上下文和企业治理

### 2.1 对照总表

| 五部分 | AutoGen | Semantic Kernel | Azure AI Agent Service |
| --- | --- | --- | --- |
| 上下文 | 通过代理消息流与角色协作维护上下文 | 通过 Kernel、提示模板、函数参数精细控制上下文 | 通过 Thread/Message 在服务端持久化会话上下文 |
| 工具 | 每个代理可挂不同工具，协作调用灵活 | 插件体系完整（本地函数、提示函数、连接器） | 提供托管工具能力并可接入 Azure 生态 |
| 记忆 | 通常自定义记忆策略，自由度高 | 记忆抽象和连接器更工程化 | 会话记忆天然具备，长期记忆常配合外部知识源 |
| 规划 | 多代理编排能力突出（群聊、轮转、路由） | 流程化编排稳健，适合业务系统落地 | 偏执行与托管，复杂规划常由上层框架实现 |
| 反馈闭环 | 事件流便于插入评估器和自定义控制 | 函数链上可做校验、重试、评分 | 监控、审计、治理友好，适合线上持续优化 |

### 2.2 每个框架的“最强杠杆点”

#### AutoGen

- 在“规划”上杠杆最大：适合研究多代理如何分工、交接、协作。
- 在“上下文”上依赖消息机制：上下文天然体现在谁对谁说了什么。
- 适合场景：POC、复杂多代理实验、策略探索。

#### Semantic Kernel

- 在“工具 + 记忆”上杠杆最大：插件和函数编排非常适合工程化开发。
- 在“上下文”上可控性高：你可以精细决定传入什么信息。
- 适合场景：企业应用开发、可维护代码、可测试流程。

#### Azure AI Agent Service

- 在“上下文托管 + 反馈治理”上杠杆最大：线程化会话和平台观测能力更强。
- 在“工具”上平台化：更方便接入企业级能力并统一管理。
- 适合场景：生产部署、合规要求高、需要统一运维监控。

## 3. 用一个统一场景看三者如何落地

场景：构建“企业数据分析助理”，用户提出“请分析上周华东区域销售异常并给出行动建议”。

### 3.1 五部分拆解（与框架无关的通用流程）

1. 上下文：读取用户问题、组织权限、时间范围、业务口径。
2. 工具：调用数据仓库查询、异常检测函数、报告生成工具。
3. 记忆：保存用户偏好（报表格式、指标优先级）和历史结论。
4. 规划：拆解为“取数 -> 分析 -> 解释 -> 生成建议 -> 输出报告”。
5. 反馈闭环：记录用户是否采纳建议，回写评分并调整下一次策略。

### 3.2 三框架落地方式差异

#### 用 AutoGen 实现时

- 把流程拆给多个代理：取数代理、分析代理、审查代理、汇总代理。
- 重点价值：多代理分工与协作策略快速迭代。
- 风险点：如果缺少统一约束，代理间对话会变长、成本上升。

#### 用 Semantic Kernel 实现时

- 把查询、分析、报告封装成插件函数，用编排流程串起来。
- 重点价值：模块清晰、可维护、可测试。
- 风险点：过度抽象会增加前期设计复杂度。

#### 用 Azure AI Agent Service 实现时

- 用线程管理用户会话和任务进度，接入平台工具与企业服务。
- 重点价值：托管、治理、监控、企业集成。
- 风险点：复杂自定义策略通常仍需结合 SK/AutoGen。

## 4. 选型建议：按“你当前阶段”来选

### 4.1 学习和验证阶段

优先：AutoGen

原因：

- 更容易观察多代理行为
- 更适合快速做策略实验

### 4.2 应用开发阶段

优先：Semantic Kernel

原因：

- 插件化和函数化清晰
- 更利于团队协作、测试、维护

### 4.3 生产部署阶段

优先：Azure AI Agent Service（可结合 SK/AutoGen）

原因：

- 托管运行、会话状态管理
- 企业级安全、监控、治理能力更强

### 4.4 常见组合路线（推荐）

1. 用 Semantic Kernel 构建核心业务代理。
2. 需要复杂多代理协作时，引入 AutoGen 进行策略层编排。
3. 需要企业级上线时，接入 Azure AI Agent Service 做托管与治理。

## 5. 实操检查清单：五部分是否真的落地

### 5.1 上下文检查

- 是否只传递当前任务相关信息？
- 是否明确了约束（权限、成本、时限）？
- 是否能解释“为什么做出这个回答”？

### 5.2 工具检查

- 每个工具输入输出是否清晰？
- 是否有失败处理与超时机制？
- 是否记录了工具调用日志用于排障？

### 5.3 记忆检查

- 记忆是否分短期和长期？
- 写入前是否做了价值判断（不是所有内容都存）？
- 是否有过期/纠错机制避免记忆污染？

### 5.4 规划检查

- 是否定义了任务完成条件？
- 是否有重试、降级、回滚策略？
- 多代理协作是否有明确交接规则？

### 5.5 反馈闭环检查

- 是否定义了可量化指标（成功率、时延、成本、满意度）？
- 是否把评估结果真正用于下一轮策略更新？
- 是否区分了“重试”与“优化”？

---

## 6. 一句话总结

AI 代理框架的价值，不是“能调用大模型”这件事本身，而是把以下五件事工程化：

- 上下文：让模型在正确边界内思考
- 工具：让模型具备执行力
- 记忆：让系统跨回合持续一致
- 规划：让复杂目标可被分解和完成
- 反馈闭环：让系统上线后持续改进

真正成熟的代理系统，通常不是只选一个框架，而是在这五部分上做最合适的组合。
