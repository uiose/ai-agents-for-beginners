## 1. 先立框架：什么是 AI Agent

AI Agent 是一个系统：

- 以 LLM 作为推理核心
- 让 LLM 能访问工具与知识
- 在环境中执行动作并改变环境状态

可记为：

> Agent = LLM + Tools + Knowledge/Memory + Runtime + Environment

### 1.1 系统视角的三个基础组件

从课程定义看，至少要能分清：

- 环境（Environment）：任务发生的业务空间（如旅游预订系统）
- 传感器（Sensors）：读取环境状态与反馈（如库存、价格、状态）
- 执行器（Actuators）：对环境做动作（如下单、取消、修改）

这一步很重要，因为它提醒我们：Agent 不是“一个模型 API 调用”，而是“一个能感知并作用于环境的系统”。

---

## 2. Agent 类型速记（课程视角）

第 1 课列出的常见类型可以压缩为：

- 简单反射型：按规则即时响应
- 基于模型反射型：结合世界模型状态决策
- 基于目标型：围绕目标规划步骤
- 基于效用型：在多目标间做收益权衡
- 学习型：基于反馈持续优化
- 分层型：高层拆任务、低层执行
- 多代理系统：多个代理协作或竞争

学习时不必死记定义，重点是识别你手上的业务更像哪一类决策问题。

---

## 3. 何时该用 Agent（而不是普通 Prompt）

第 1 课给了三个非常实用的判断标准：

- 开放性问题：步骤无法完全硬编码
- 多步骤流程：需要多轮决策和工具调用
- 可随时间改进：系统需要基于反馈持续变好

反过来说，如果问题是固定模板、单步可解、无需反馈闭环，通常不必上 Agent。

---

## 4. 回到 notebook：这段代码在做什么

示例核心结构：

```python
class DestinationsPlugin:
    @kernel_function(description="Provides a random vacation destination.")
    def get_random_destination(self) -> Annotated[str, "Returns a random vacation destination."]:
        ...
```

这段代码的准确理解是：

- `DestinationsPlugin` 是工具集合（plugin）
- `get_random_destination()` 是被暴露给模型可调用的函数（tool）

所以这里不是“只定义了一个函数”，而是“定义了插件并注册了可调用工具”。

---

## 5. 最小可运行 Agent 流程

一个最小流程可以写成：

1. 接入 LLM
2. 定义工具
3. 将工具注册到 Agent Runtime
4. 接收用户自然语言请求
5. 模型判断是否调用工具
6. Runtime 执行工具并回传结果
7. 模型组织最终自然语言回复

一句话：

> 用户提需求 -> 模型选工具 -> 框架执行 -> 模型整理结果

---

## 6. 职责边界：谁负责什么

在 Tool Calling 里，职责要分开看：

- LLM：理解意图、选择是否调用、决定调用哪个工具
- 框架/Runtime：提供工具描述、执行真实函数、回填结果
- 工具函数：完成特定能力（查、算、写、调接口等）
- Agent：将以上部件组织成可持续运行的任务单元

核心记忆句：

> LLM 负责“决策调用”，程序负责“真实执行”。

---

## 7. 模型为什么“知道”有哪些工具

模型不会直接读取你的本地 Python 代码。

真实过程是：

1. 代码中注册工具
2. 框架将工具转成结构化描述（schema）
3. 请求模型时，把“工具说明 + 对话上下文”一起发送

通常包含：

- 工具名
- 描述
- 参数 schema
- 返回语义

所以模型知道工具，是因为 Runtime 把“说明书”交给了模型。

---

## 8. 模型如何决定要不要调用工具

常见依据：

- 用户当前请求意图
- system/instructions 约束
- 工具名与描述语义匹配度
- 参数是否可满足当前任务

示例：

- “帮我推荐一个旅行地点”更可能触发 `get_random_destination`
- “什么是旅行规划”更可能直接文本回答

因此，是否调用是上下文推断，不是硬编码 if-else。

---

## 9. 一次 Tool Call 的完整链路

1. 注册工具
2. 发送工具 schema 给模型
3. 模型返回 tool call 请求
4. Runtime 执行本地/远程函数
5. 执行结果回传模型
6. 模型生成最终回复

关键事实：

- 模型不会直接执行 Python
- 执行权始终在宿主程序

---

## 10. 为什么工具描述质量决定调用质量

模型选工具时主要依赖描述文本与参数定义。

高质量描述应满足：

- 名称清晰（做什么一眼可见）
- 描述准确（不要夸大能力）
- 参数明确（减少歧义）
- 粒度合适（避免过粗或过碎）

如果描述模糊，常见后果是“不调用该调的工具”或“误调相似工具”。

---

## 11. 工具变多后，为什么成本和错误率会上升

工具多意味着每轮上下文中“能力说明书”更长，结果通常是：

- token 消耗增加
- 延迟上升
- 成本提高
- 工具选择歧义增加

这不是框架缺陷，而是上下文带宽与决策复杂度的自然代价。

---

## 12. 成熟系统如何控 Tool 数量

常见策略：

- 按场景暴露工具：一次只给当前任务相关工具
- 先路由再注入：先判意图，再挂载对应工具组
- 精简描述：保留辨识信息，减少冗余 token
- 高层封装：对外暴露能力级工具，内部再拆细粒度步骤
- 动态加载：按会话阶段启用/卸载工具
- 先检索后提供：从大工具库中筛选 top-k 相关工具给模型

## 14. 最终结论（概念层 + 机制层）

结合 README 与 notebook，可得到一组稳定结论：

- Agent 是系统，不只是一次 LLM 调用
- Tool 是 Agent 执行能力的桥梁
- 模型通过 schema 感知工具，不直接读本地代码
- 模型负责决策调用，Runtime 负责真实执行
- 工具设计质量直接影响调用正确率
- 工具数量需要治理，否则会引起上下文膨胀与成本上升

如果只保留一句话：

> 模型决定“该调用什么”，框架完成“真正执行并返回结果”。
